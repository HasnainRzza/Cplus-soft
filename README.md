# CV-Aware Conversational RAG API

A production-grade Retrieval-Augmented Generation (RAG) system built with FastAPI and LangChain. The application supports general-purpose conversational AI as well as deep, context-aware question answering over user-uploaded CV/Resume documents. The architecture is designed around the principles of non-blocking I/O, semantic caching, and structured observability.

---

## Architecture Overview

The system is composed of six independent, cohesive layers that work in concert.

```
User Request
     |
     v
[FastAPI Application]
     |
     +---> POST /upload  -->  [BackgroundTask: PyPDF Loader -> Sentence Transformers -> FAISS Index]
     |
     +---> POST /chat   -->  [Semantic Frequency Cache]
                                    |
                       +------------+------------+
                       |                         |
                   CACHE HIT                 CACHE MISS
                   (freq >= 3)          [LangChain Agent Executor]
                       |                    |           |
                Stream Cached          search_cv()    Groq LLM
                Response               [FAISS DB]   (qwen3.6-27b)
                                           |           |
                                           +-----------+
                                                 |
                                      [Stream tokens via SSE]
                                                 |
                                    [asyncio.Queue: Log Event]
                                                 |
                                    [Background Flush: app.jsonl]
```

## Key Features

### Agentic Tool Calling
The core of the system is a LangChain tool-calling agent. When a user asks a question, the agent autonomously decides whether to invoke the `search_cv` retrieval tool against the FAISS vector database, or to answer directly from its conversational context. This removes the need for manual routing logic and produces significantly more accurate responses.

### Asynchronous CV Processing
The `POST /upload` endpoint delegates all CPU-bound work (PDF parsing, text chunking, and FAISS index construction) to FastAPI's built-in `BackgroundTasks`. The endpoint returns a `202`-style acknowledgment immediately, ensuring the user is never blocked waiting for embeddings to be generated.

### Semantic Frequency Cache
Before any query reaches the Groq API, it passes through a custom-built semantic cache. The cache maintains an in-memory list of past query embeddings. Cosine similarity (computed via NumPy) is calculated between the incoming query vector and all stored vectors. If a match above the 0.85 similarity threshold has been seen three or more times, the cached response is returned instantly without making any external API calls, reducing latency to near-zero and eliminating token cost for frequent questions.

### Streaming Responses
The `/chat` endpoint wraps the agent's output in a FastAPI `StreamingResponse`. Using LangChain's `astream_events` interface, tokens are streamed to the client as they are generated by the LLM, providing a responsive, real-time user experience rather than a long wait followed by a bulk response.

### Sliding Window Conversational Memory
The agent maintains per-session chat history using `RunnableWithMessageHistory`. A sliding window of the last 10 messages is enforced on each request to prevent context overflow and control token consumption while retaining meaningful conversational context.

### Asynchronous Structured Logging
All system events (cache hits, cache misses, CV uploads, and errors) are pushed into an `asyncio.Queue` in a non-blocking manner. A background coroutine, started on application startup, wakes up every 60 seconds and flushes the entire queue to disk as structured `JSON Lines` (`app.jsonl`). This pattern ensures logging never adds latency to the request-response cycle.

## API Reference

### `POST /upload`
Accepts a PDF file and initiates asynchronous document processing.

**Request:** `multipart/form-data` with a `file` field containing a `.pdf` document.

**Response:**
```json
{
  "message": "CV upload received. Processing in background..."
}
```
The response is returned immediately. The FAISS index is built in a background thread.

---

### `POST /chat`
Accepts a user query and returns a streamed, contextual response.

**Request Body:**
```json
{
  "session_id": "user-abc-123",
  "query": "Summarize the candidate's experience with distributed systems."
}
```

**Response:** A plain-text stream (`text/plain`) of tokens delivered progressively as the LLM generates them. If the query is a cache hit, the cached response is returned as a single instant stream.

## Configuration

All tunable parameters are centralized in `config/config.py`:

| Parameter | Default | Description |
|---|---|---|
| `CACHE_SIMILARITY_THRESHOLD` | `0.85` | Cosine similarity score required to match a cached query |
| `CACHE_FREQ_THRESHOLD` | `3` | Number of similar queries before a response is cached |
| `LOG_FLUSH_INTERVAL_SEC` | `60` | How often the background worker writes logs to disk (seconds) |
| `FAISS_INDEX_PATH` | `./faiss_index` | Filesystem path for the persisted FAISS index |
| `LOG_FILE_PATH` | `./logs/app.jsonl` | Output path for structured JSON event logs |

## Setup and Installation

**1. Clone the repository and create a virtual environment.**
```bash
git clone https://github.com/HasnainRzza/Cplus-soft.git
cd Cplus-soft
python -m venv venv
venv\Scripts\activate
```

**2. Install all dependencies.**
```bash
pip install -r requirements.txt
```

**3. Create a `.env` file in the project root with your Groq API key.**
```
GROQ_API_KEY=your_groq_api_key_here
```

**4. Start the development server.**
```bash
python -m src.main
```

The server will be available at `http://localhost:8000`. Interactive API documentation is available at `http://localhost:8000/docs`.

## Design Decisions

**Why FAISS over a hosted vector database?**
For a self-contained demonstration, FAISS provides a performant, zero-infrastructure vector store with full local persistence. It can be swapped for a hosted solution (Pinecone, Weaviate) by modifying only `src/vectorstore.py`.

**Why an in-memory cache over Redis?**
The semantic cache is intentionally implemented without external dependencies to keep the deployment footprint minimal. The architecture is designed so that the `FrequencySemanticCache` class in `src/semantic_cache.py` can be replaced with a Redis-backed implementation with no changes to the agent or API layer.

**Why asyncio.Queue over a message broker?**
For the scale of this application, a built-in asyncio queue provides genuine non-blocking event dispatch without the operational overhead of RabbitMQ or Kafka. The 60-second batch-flush pattern mirrors standard write-behind caching strategies used in production telemetry pipelines.
